# weaponx-plan-2026-07-02-0630

## Summary (read this first)

Built the piece that was missing when you asked "could Weapon X actually build a complex
app": a new skill, `weaponx-plan`, that takes a genuinely large, open-ended idea and turns
it into something the existing loop can actually execute, instead of asking one 4-cycle
retry loop to somehow converge on something that's really many tasks' worth of work.

**What it actually does:** runs a real planning pass (using the same planning skills you'd
use by hand — office-hours to clarify the idea, autoplan or plan-eng-review to produce a
real architectural plan), breaks the approved idea into a sequence of normal-sized stages
ordered by real dependency (never in parallel — that's still explicitly not built), asks
you to approve the whole stage list exactly once — with the total scope spelled out in
plain numbers so you're not signing off on more than you realize — and then, only after
that one approval, runs each stage through the exact same five-move loop that's been
proven all night, one at a time, stopping the whole sequence cold if any stage fails
rather than pretending a failed dependency succeeded.

**The core loop itself was never touched.** `weaponx-plan` doesn't reimplement discovery,
generation, verification, or persistence — it calls the real `/weaponx` skill for each
stage, the same way you'd type it yourself, and reads that stage's own trace file
afterward to know what happened. Confirmed by three separate evaluator checks across two
cycles that `weaponx/SKILL.md`, `weaponx-discover/SKILL.md`, and both evaluator
definitions have zero changes.

**It took two real rejection rounds to get right, same as the last two things built
tonight:** the first version only described the plan-tracking format instead of showing
it working, and it accidentally referenced one of this project's own file names inside the
otherwise-portable skill file — a fork wouldn't have had that file. Separately, one
evaluator flagged (not a reject, but a real concern) that approving a long stage list
doesn't automatically mean a human has mentally added up the total cost — so the approval
step now has to state the total stage count and the worst-case cumulative budget in plain
numbers, not just list the stages and hope you do the math.

**What this does NOT mean yet:** this has never been run against a real idea. Everything
checked tonight is static — does the skill's own logic hold together, does it correctly
call the real tools it says it calls, is the one approval gate actually airtight in how
it's worded. The actual first live test — pointing this at a real project and watching it
plan, get approved, and execute stage by stage — hasn't happened. That's the honest next
step, not another review cycle.
